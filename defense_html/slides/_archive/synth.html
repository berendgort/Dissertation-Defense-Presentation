S35 · 35-node scenario
9.5%
vs 55.5% at S18
same architecture
same base model
Root cause: context bloat at scale. The architecture is correct. The constraint is the LLM.
Why the architecture is still the right answer
01
LLMs will handle larger contexts
Context windows are growing fast and reasoning quality at scale is improving every model generation. S35 is a hard case today; it will not be a hard case in two years. The architecture does not need to change for the result to improve.
02
State loading is naive today
The current implementation feeds the full cluster state into the LLM context. Active research on condensed state representations, semantic filtering, and retrieval-augmented context can reduce the tokens the agent needs to reason over, without changing the PARES architecture at all.
9.5% at S35 · context bloat · LLM headroom + smarter state loading

Backup B2//AgentEdge · state staleness
B2/B8   Backup
The remedy is already in the architecture
ActSimCrit re-checks before it executes.
The Actuation agent re-queries live infra state immediately before committing actions. A stale plan is caught at the execution gate, not after damage is done.
Path to eliminating the gap
01
Faster planning closes the window
Swapping the LLM planner for a classical optimization solver cut response time from 107 s to 48 s with no architecture change. Newer, faster models will close it further.
02
Smaller context reduces the risk
Today the agent loads the full cluster state. Filtering to a relevant subset shortens planning time and shrinks the window in which the infra can drift. No architecture redesign needed.

Backup · slide 03 motivation numbers · sources

Sources for the <40% utilization and edge-headroom claims

Claim: <40% average utilization
Google Borg study (Reiss et al., 2012); Microsoft Azure capacity studies; EC2 usage analyses. Consistent finding: cluster utilization in production data centers rarely exceeds 40% on average due to headroom reservation for burst tolerance.
Claim: edge headroom evaporates under bursts
Edge nodes have 1 · 2 orders of magnitude less compute than cloud clusters. Burst tolerance headroom that is adequate at cloud scale is insufficient at the edge · reactive scheduling has no room to maneuver.
