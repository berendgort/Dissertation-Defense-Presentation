B1/B18   Backup
“Planning runs for tens of seconds, sometimes hundreds — up to 700 s in our runs. Isn’t the plan already stale by the time it executes?”
Honest answer: yes, long planning windows open a drift window. That drift window is contained by design, bounded by scope, and actively engineered down.
01
Strategic orchestration, not reactive control · AgentEdge governs consolidation, SLA protection, and energy — decisions that already operate on minutes-to-hours horizons today, not sub-second packet scheduling. Drift within a planning window is a second-order effect on that cadence.
02
Drift is reconciled at execution, per call · Infra Action runs the validated plan as sequential API calls. On any failed call it transitions from script–executor to single-agent problem solver, diagnoses against the live state, and issues corrective actions in that same call — no bounce back to ActSimCrit. Unsafe-action containment stays 100 % across 630 runs. Drift costs extra calls, never safety. (See B12.)
03
Runtime is engineering, not architecture · A drop-in ILP action-selector already cuts planning in half on the hardest scenario — one lever of many (specialised small models, hierarchical planning, structured context, cached sub-plans, speculative decoding). None changes the AgentEdge contract.
04
Model scaling is a tailwind · We ran Qwen3-235B (Q1 2025); agent runtime scales with parameter count. Today’s ~30 B-class models already match or exceed that intelligence — roughly 7× fewer parameters, ~7× faster at equal or better quality. The clock on this weakness is running against the question. See B2.
