Adaptive periodicity in three steps

01
Period Detection
Dominant period changes, AERO follows it
autocorrelation on each window → updated dominant τ
02
Adaptive forecasting
Adjust to one valid τ*, reshape once, predict once

03
Layer caching
Seen this τ* before? Reuse the layers.
cache keyed by adjusted period τ* reuses the forecasting layers when the same pattern returns
AERO · detect τ → adjust to τ* → forecast → cache

Rationale
23/54   OmniFORE
Design rationale

Generalization comes from the training set

Pain point
Workload heterogeneity
bursty · steady · periodic · same system, very different shapes
Scale varies 100×
big services drown out small ones during training
Standard tuning overfits
settings picked on known services · fail on new ones
Design decision
Group similar workloads, pick one of each
every workload type is seen during training, not just the common ones
Put every service on the same scale
rescale each service so big and small workloads carry equal weight in training
Tune for services we haven't seen
score the model on held-out services, not on the ones it was trained on
OmniFORE · rationale · curate, normalize, tune for transfer

25/54   OmniFORE
Phase 1 · Curate · S₁ → S₄

Designing the training set

groups → balanced mix → attention head sees them all
S₁ Fingerprint each trace
Compress each trace into a shape descriptor.
S₂ Cluster into families
Same shape → same group.
S₃ Tag every trace
Stamp each trace with its group label.
S₄ Pick a balanced mix
Sample proportionally from each group.
Why this training set?
Current forecasters forget past points; windows roll past, context dies with them. A balanced catalogue only pays off if the model can remember what it saw.
Attention → longer memory
Attention lets OmniFORE look back across the entire window and weight what matters: bursty spikes, steady drift, or periodic tick, giving it the memory generalisation needs.

Solution 2 · stage S₇
27/54   OmniFORE
Phase 3 · Tune · stage S₇

Tuning so it transfers

How we search for good settings
We don't try every combination. A smart search builds a guess of where good settings live, tries the most promising spot next, then updates the guess.
What "good" means here
A setting is good if the model does well on services it has never seen before.
We hold a batch of services out of training. After each try, we check how close the forecast is to the truth on those · averaged across all of them. That average is our score.
Usual way
Score on services the model trained on. Model gets great at those · bad at new ones.
Our way
Score on services held out of training. Only settings that transfer win.

Q·10 · Can AERO forecast other 6G signals?
Yes. any univariate time series with learnable periodicity.
The architectural contract is: observe → attend over periodicity → forecast. It does not depend on the signal being CPU usage.
Candidate signals in a 6G stack
RAN · PHYSICAL
Physical resource-block (PRB) utilisation per cell · MCS distribution · active connections
CORE · CONTROL
UPF throughput · session setup rate · slice-level QoS margin
EDGE · APPLICATION
Service request rate · inference latency percentiles · energy per request
USER PLANE
Downlink/uplink byte rates per UE class · mobility-driven handover rate
Caveat: non-periodic signals (e.g. pure mobility noise) need a companion module; AERO's inductive bias is periodicity. OmniFORE is the candidate for those cases.
Any periodic time series · RAN / core / edge / user plane · OmniFORE for non-periodic
